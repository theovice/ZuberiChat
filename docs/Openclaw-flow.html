OpenClaw Runtime Flow

How a task moves through ClawBot → Runtime → LLM → Tools → Response
Example Task Input
User → ClawBot: "Find any open P0 bugs for Project X, summarize them, and post the summary in #triage."
Interface
ClawBot — chat surface
Where messages enter and results return. Not an agent — just the channel.
Receives raw user message from Telegram, Discord, Slack, or CLI
Identifies user + session context (who is this, what project, what permissions)
Hands task + context to the Runtime — its job ends here until the final answer comes back
Posts final output back to the originating channel
▼
task + session context passed to Runtime
Runtime
OpenClaw Runtime — the agent engine
The orchestration core. It does not think — it enables and enforces.
Loads the relevant Skills (playbooks / SOPs) for this task type
Registers available Plugins (local tools) and MCP servers (remote tools)
Applies the permission/allow-deny policy: what tools can run, with what scope
Sends the full context (task + skills + available tools + policy) to the LLM
Manages the loop: receives LLM tool-call requests → executes them → feeds results back → repeats until done
▼
task + skill guidance + allowed tools sent to LLM
LLM
LLM — the planner and decider
The only component that "thinks." Everything else executes what it decides.
Reads the task, skills, and tool list — decides the next step
Outputs structured tool-call requests (not free text) — e.g., which tool to call, with what arguments
Reads back tool results and decides: call another tool, or produce final answer?
Drafts the human-readable summary / confirmation when the task is complete
Skills guide how it approaches the task — the LLM still decides what to do step by step
▼
structured tool-call request (tool name + args) sent to Runtime
Tool Layer
Tool Execution — run by the Runtime, requested by the LLM
Three types of tools, all reaching the LLM the same way — as results fed back into the loop.
Skills
Playbooks / SOPs. Not a tool the LLM calls — guidance loaded into its context before it starts.
Example instructions injected:
When asked for P0 bugs:
1) query the tracker
2) group by component
3) bullet format summary
4) post to #triage w/ template
Plugins
Local, in-process tool functions. Fast, no network hop. Installed code the Runtime can call directly.
Example calls:
jira.searchIssues() slack.postMessage() formatter.makeTriageSummary() fs.readFile() git.commit()
MCP Servers
Standardized external tool providers over the MCP protocol. Same idea as plugins — but remote, isolated, and swappable.
Example calls:
mcp://github/search_issues mcp://linear/list_issues mcp://confluence/get_page mcp://browser/navigate
▼
tool results returned to LLM — loop repeats until task is complete
Output
ClawBot → User — task complete
LLM produces final answer. Runtime hands it back to ClawBot. ClawBot posts to the originating channel.
Summary of P0 bugs posted to #triage in Slack
Confirmation message sent back to the user who requested it
Gateway log records: which tools were called, results, duration, any errors
The Micro-Loop
What one full cycle looks like inside the Runtime ↔ LLM back-and-forth
L
LLM → Runtime (tool request)
Decides first action. Outputs a structured call:
jira.searchIssues({ project: "X", priority: "P0", state: "open" })
R
Runtime → Plugin (executes)
Checks permission, runs the plugin, gets back a list of issues.
→ Returns: [ {id: "X-101", title: "Auth fails on login"}, {id: "X-102", ...} ]
L
LLM → Runtime (next tool request)
Reads the issue list. Decides to format it now:
formatter.makeTriageSummary({ issues: [{...}, {...}] })
R
Runtime → Plugin (executes)
Formats the issues using the triage template defined in the Skill.
→ Returns: "**P0 Bugs — Project X**\n• X-101: Auth fails on login\n• X-102: ..."
L
LLM → Runtime (final action)
Has everything it needs. Posts to Slack:
slack.postMessage({ channel: "#triage", text: "**P0 Bugs — Project X**\n..." })
R
Runtime → Plugin (executes)
Posts to Slack. Gets confirmation back.
→ Returns: { ok: true, ts: "1234567890.123" }
✓
LLM → Final Output
No more tool calls needed. Produces human-readable confirmation for the user. ClawBot posts it back to the originating channel.
→ "Done — found 2 open P0 bugs for Project X and posted the summary to #triage."
